# Stay-Active Rule

These rules apply to every **loop** and **exec** workflow, in every mode (`general`, `fast`, `skill`), on every platform. They govern the **main agent / controller** and every subagent it spawns.

---

## Rule 1 — Do not stand aside early. Stay active.

The agent stays **actively engaged from the first step to the terminal step**. Delegating work, starting a command, or finishing one iteration is **never** a reason to go quiet.

**Never:**
- End the turn, hand back to the user, or declare the workflow finished while a subagent, background command, or dispatched sub-main agent is still in flight.
- Idle, "stand by", or wait passively for something to happen.
- Ask the user to report back when a build/test/CI/subagent finishes, or to "tell me when it's done".
- Treat a delegation as the end of a step — the step ends when its **result has been observed and recorded**, not when the work was handed off.

**Always:**
- After every delegation, either do the next non-blocking work (observe, measure, verify, ledger) or actively watch the in-flight work through to its result.
- Drive the loop / execution forward on your own initiative until a terminal condition is reached.

**The turn may be yielded only when one of these is true:**
1. A terminal **exit condition** fired (loop), or every step completed (exec).
2. A **mechanical halt** was recorded as terminal — a workflow-defined data-integrity or duplicate-work-safety condition (corrupted or ambiguous durable state, a failed integrity check, a provably unsatisfiable measurement contract). A feasibility or difficulty judgment — the target looks unreachable, sits above a published ceiling, or costs more compute than it seems worth — is **never** a mechanical halt and never a reason to yield: record it, then re-plan and continue autonomously (switch strategy class, restructure the step, re-estimate and respawn).
3. The **approval gate** is in Mode 1 (plan-only) per `_lib/approval_gate.md`.

Anything else — including "the subagent is running" or "the tests are still going" — is **not** a valid reason to stop. The workflow runs autonomously: difficulty, doubt, or a recorded blocker is never a reason to pause for human input mid-run (the only mid-run pause that remains is `_lib/approval_gate.md` Mode 2's gate on outward-facing/irreversible actions such as a git push).

> **Hard enforcement (optional):** on every platform these rules are guidance, not a deterministic guarantee. On Claude Code, teams can back the stay-active rule with a platform-level guard — native `/goal` (per its official docs, "a wrapper around a session-scoped prompt-based Stop hook"; set the loop's exit OR-set as the condition), a custom Stop hook, or the official `ralph-loop` plugin (per the official hooks guide, hooks give deterministic control instead of relying on the model to comply). Two guards are mandatory when doing so, and both apply to **loop workflows only** (exec has no scratch file or [re-entry prompt] to re-feed): **(a)** the guard's re-fed payload must be the **[re-entry prompt]** persisted in the loop's scratch file (defined in `_lib/loop_control.md` §Re-Entry Prompt) — never the raw original request, which would replay the whole pre-loop spec design (e.g. the general loop's analyst-panel steps) every cycle. (Stock `ralph-loop` deliberately re-feeds the same unchanged prompt and keeps state in files/git — it proves the Stop-hook re-feed mechanism; the resume-aware payload is this pack's addition on top.) **(b)** the guard must **disarm as soon as any exit condition is recorded in the [loop ledger]** — otherwise it fights the `max_iterations` unconditional cap, the one stop nothing may override. Native constraints: Claude Code force-releases a Stop hook after 8 consecutive blocks without progress (raise via `CLAUDE_CODE_STOP_HOOK_BLOCK_CAP`) — a cap independent of the loop's own `max_iterations`; and `/goal` and Stop hooks are unavailable when `disableAllHooks` is set at any settings level or `allowManagedHooksOnly` is set in managed settings.

## Rule 2 — Every wait arms two wake triggers, through two different mechanisms. Wake safety is per-wait.

Sometimes a step genuinely must wait: a long build, a background process, a remote CI run, an external event. A wait is allowed **only** when the agent is guaranteed to wake up again. A single wake path is a single point of failure — if it never fires, the agent sleeps forever.

**A wait may begin only when durable state was just reconciled, the awaited condition is still false, and one live event trigger plus one live bounded time trigger are armed. This holds per-wait, not per-task: a trigger that already fired is consumed and cannot protect a later wait.**

- **Trigger A — event-driven.** Fires when the awaited thing actually changes state. *(Claude Code: a `Monitor` watch on the condition, or the completion notification of a background `Bash` task / subagent. Codex · VS Code Copilot: a blocking foreground call with its own hard timeout.)*
- **Trigger B — time-driven fallback.** Fires on a timer regardless of whether the event ever arrives. *(Claude Code: `ScheduleWakeup`, a `CronCreate` tick, or a bounded re-check loop. Codex · VS Code Copilot: a bounded polling loop that re-reads real state on an interval.)*

**Requirements:**
1. **Two different mechanisms — not one mechanism twice.** Two timers are one mechanism; an event watch plus a timer are two.
2. **Follow the wait sequence: reconcile → arm both → reconcile again → wait.** Arm both before the wait begins, never after — an unarmed fallback cannot rescue a wait already in progress. The second reconciliation closes the race where the work completes between the first check and trigger registration: if it now shows the condition true, consume the triggers and continue instead of waiting. *(Codex · VS Code Copilot: both triggers are synchronous — a bounded blocking call or poll loop — so this collapses to re-reading real state immediately before each bounded call or poll iteration.)*
3. **Two timer concepts, one bound: a renewable watchdog tick under an immutable absolute deadline.** The time-driven trigger is a **watchdog tick** — it wakes the agent to reconcile and may be re-armed on every wake. The **absolute deadline** is set once, before the wait begins, and re-arming ticks never extends it — otherwise a single step could wait forever despite the workflow's caps. Declare the deadline up front: default **1 hour**, or a larger bound declared before the wait for known-long work (training, full benchmark runs). A pre-spawn time estimate may inform the declared bound, but is never grounds to kill work early — model duration estimates are unreliable. The deadline covers one wait's re-armed generations for the **same awaited work**; a new wait for different work gets its own deadline.
4. **On deadline expiry, treat it as a hard blocker.** Reconcile once more, persist the pending-wait record (point 6), then record it and recover autonomously per Rule 4 point 4 — never sleep unbounded, and never silently kill-and-continue. Where the platform exposes a safe stop for the stuck work (e.g. Claude Code's `TaskStop`), the recovery may stop it **after** the record is persisted, discarding only a repeat-safe, uncommitted tail.
5. **Whichever fires first wins — and is thereby consumed.** On wake, stand down the other trigger and **re-verify the real condition by reading actual state** (exit code, file, process status) — a trigger firing, or any notification, is a hint to look, not proof the work finished. A stale notification from an earlier wait generation gets the same treatment: reconcile, then ignore it. Before waiting again, retire the old generation and arm a fresh live pair (under the same absolute deadline while still awaiting the same work).
6. **Persist a pending-wait record before the wait begins** — never deferred to the later Reflect/Ledger or report step: wait generation (incremented each time the pair is re-armed for the same awaited work), awaited work/artifact identities, start time + absolute deadline, last reconciliation result, and trigger lifecycle (live / fired / consumed / replaced) where the platform exposes it. Write it to scratch state (loop: a note beside the scratch ledger, not a ledger line; exec: a scratch note) — never a repo file. After the wake, record the completed wait (what was awaited, both triggers, which fired, duration) in the [loop ledger] entry / [execution report] as before.
7. **On every wake or resume, inspect authoritative completion metadata first.** If the awaited work is complete, accept its output only when its work identity matches the pending-wait record **and** the step's own verifier passes — then continue immediately and commit it (loop: per the iteration-commit rule, write-guard included; exec: record it in [execution report]). On an identity mismatch or failed verification, discard the cached output and fall back to the repeat-safe replay (loop: re-run that pass's Act step; exec: safe re-execution per the plan) — never accept stale output or duplicate a committed side effect. If the work is incomplete, retire the old wait generation and arm a fresh pair before waiting again (point 5). This is the level-triggered controller pattern — observed state governs; notifications are only optimizations ([Kubernetes controllers](https://kubernetes.io/docs/concepts/architecture/controller/), [watch recovery](https://kubernetes.io/docs/reference/using-api/api-concepts/)).

**Design the wait so that both triggers failing is impossible, not merely unlikely.** This protocol engages only when a wait is genuinely unavoidable: if you cannot name two distinct mechanisms for a given wait, restructure the step so it does not wait at all — poll actively instead.

### Durable in-flight protocol

Every workflow governed by this file must bind exactly one **canonical durable in-flight record** before its first delegation or yield. The workflow specification names the path; if it has no more specific record, use a single workflow-owned file beside its ledger or execution report. Never split authority between chat context, a worker transcript, trigger metadata, and several scratch files. The record is the source of truth after interruption and contains:

- a monotonically increasing `generation_high_water` that is never reset or reused;
- an `active_waits` map keyed by generation, so parallel delegations remain independently recoverable;
- for each generation: owner step/round, a stable launch idempotency key and launch receipt, awaited artifact or condition, worker/process handle (once known), start time, absolute deadline, event mechanism plus stable trigger key/receipt/handle, fallback mechanism plus stable trigger key/receipt/handle, resume action, `setup_complete`, and status;
- the wake that claimed the generation, the real-state verification result, terminal outcome, and cleanup status.

Coordinate every read/replace with one stable sidecar lock whose pathname is fixed for the workflow and whose file is never renamed or deleted; never lock the canonical record inode that atomic replacement supersedes. Update the record atomically while holding that lock (write a complete replacement, flush it, then rename it over the old file; use equivalent transactional storage when a file is not available). Apply the following lifecycle to **every delegation and every yield**, including delegations whose result is expected immediately:

1. **Prepare before delegation.** Under the sidecar lock, increment `generation_high_water`, insert that generation into `active_waits` with status `PREPARED` and `setup_complete: false`, and durably persist all fields known before launch: a stable `launch_key` derived from workflow/run identity + owner step/round + generation; stable `event_trigger_key` and `fallback_trigger_key` derived from workflow/run identity + generation + the respective mechanism; awaited artifact; start time; absolute deadline; both planned mechanisms; and resume action. Only then launch or delegate, passing `launch_key` to the platform when supported. Persist the returned launch receipt and worker/process handle immediately in the same generation before any other action; a launch failure becomes that generation's terminal outcome. On recovery from a crash between launch and receipt persistence, query the platform by `launch_key` and adopt the unique matching launch—never blindly respawn. If the platform cannot locate or adopt it and cannot prove that no launch occurred, record a hard blocker rather than risk duplicate work.
2. **Arm before yield.** Before arming, recheck under the sidecar lock that the generation remains `PREPARED`; if it is already `CLAIMED`, arm nothing, set `setup_complete: true`, and hand off to the claimant. Otherwise arm the event trigger using `event_trigger_key` where supported, then immediately persist its returned receipt and concrete handle even if the generation was concurrently claimed; recheck the claim state. Only if still `PREPARED`, arm the fallback trigger using `fallback_trigger_key` where supported, immediately persist its returned receipt and concrete handle even if concurrently claimed, and recheck again. A crash after either arm but before receipt/handle persistence must reconcile by querying the mechanism with that trigger key and adopting the unique matching trigger. It may arm a replacement only after proving no matching trigger was created; if it can neither adopt nor prove non-creation, record a hard blocker. Transition to `WAITING` and set `setup_complete: true` only if both handles are durable and the generation is still `PREPARED`. A completion arriving during setup may claim `PREPARED`; in that case the setup path must not overwrite `CLAIMED`, must not arm another trigger after observing the claim, and must not yield. It records/adopts and cancels any trigger already armed, sets `setup_complete: true`, and leaves final generation-owned cleanup to the claimant. A claimant must not finalize cleanup or remove the generation until `setup_complete` is durable. After both handles are durable, re-read real state immediately before yielding and submit a generation-tagged local wake if it is already terminal; this closes the launch/registration race. Do not yield or rely on background completion until both handles are confirmed in the record. If either cannot be armed, cancel only handles owned by this generation, persist `setup_complete: true`, and remain active with bounded foreground polling; do not enter a wait.
3. **Claim exactly once.** Every wake carries its generation. Under the same sidecar lock, it may atomically change only its matching active generation from `PREPARED` or `WAITING` to `CLAIMED`. A missing generation, a generation other than the awaited one, or a generation already `CLAIMED`/terminal is a stale or duplicate wake: record an optional bounded diagnostic counter and otherwise perform **no resume action, scoring, logging, state advance, or cleanup**.
4. **Verify real state.** The winning wake is only permission to inspect. Re-read the actual process status and exit code and validate the awaited artifact's identity and completeness; file existence alone is insufficient. Also compare the absolute deadline to the current clock. Persist the verification evidence before acting on it. If work is incomplete and the deadline has not passed, close this generation as `RECHECKED`, generation-owned-clean it, then create a new generation by repeating this protocol before waiting again. Reusing the generation would let a late duplicate claim the new wait.
5. **Finish idempotently.** Persist the terminal outcome before performing its resume action. Every resume-side effect must have a stable commit key recorded at its destination and must no-op when that key is already committed. Cancellation and cleanup may touch only the worker and trigger handles listed on or adopted into the claimed generation; record each cleanup result so recovery can retry unfinished cleanup safely without canceling a newer generation's handles. Remove the generation from `active_waits` only after its outcome and cleanup are durable; retain `generation_high_water` and any workflow-required tombstone.

On controller start, session resume, and before allocating any new generation, acquire the stable sidecar lock and reconcile the canonical record. For each `PREPARED`, `WAITING`, or `CLAIMED` generation, first reconcile its launch and both trigger keys/receipts, adopt any unique live launch or trigger and persist the adopted handles, then inspect real state, reclaim only through the transition above, complete any already-committed resume action idempotently, and finish generation-owned cleanup. Ambiguous launch or trigger creation is a hard blocker, never permission to duplicate it. Chat notifications and timers are hints; the durable record plus real state decide what happens. A recorded hard blocker here is always mechanical (data-integrity / duplicate-work safety), never a feasibility judgment: resolve it by re-planning a provably non-duplicative continuation, and only when none exists does it end the run as a recorded terminal failure (Rule 1) — never as a pause for human input.

## Rule 3 — Heartbeat every long wait or delegation.

While any wait or delegation is in flight, emit a **heartbeat** on a bounded interval: a one-line liveness note in the workflow's own record ([loop ledger] entry-in-progress, [execution report], or a progress note) stating what is in flight, how long it has been running, and when the next check is. A heartbeat is a read-and-report, not a new action — it never replaces Rule 2's wake triggers.

**Every heartbeat verifies the delegated agent is still alive — liveness is checked, never assumed:**

1. **Check the worker, not just the wait.** On each heartbeat, read the platform's authoritative status for the delegation (Claude Code: the task/subagent status or background-shell state; a raw process: PID/PGID existence **with** matching process-start identity, so a recycled PID never passes) and one observable progress signal of its output (log bytes/lines advanced, step counter, result-envelope mtime) compared against the previous heartbeat.
2. **Record the evidence in the heartbeat line** — e.g. `14:20 heartbeat: train g7 in flight 42m, pid 31337 alive (start-id match), train.log +512 lines, step 3400/20000, next check 14:30` — in the workflow's own record as above.
3. **A dead worker is acted on at the heartbeat, never at the deadline.** If the worker is provably gone (task terminal, process absent) while its wait is still in flight, treat this heartbeat as the effective wake: reconcile real state per Rule 2 point 5 — the work may have finished without the event firing — and, if it truly failed, follow Rule 4 point 4's record-and-re-plan immediately. Never leave a dead delegation "waiting" until the absolute deadline notices it.
4. **Alive-but-silent is scrutiny, not a kill.** A worker that is alive with no observable progress across consecutive heartbeats gets a deeper check (process state, device utilization, timestamp of last output) and a stall note in the record; the kill decision itself still belongs to Rule 4's run limit — silence alone is never grounds to kill early.
5. **Workers heartbeat too, where the platform's status is coarse.** A worker running long work appends its own timestamped one-line heartbeat (elapsed, launched-process PID alive, latest progress signal) to its durable result-envelope/progress path on the same bounded interval, so the spawner can distinguish a live-but-slow worker from a hung one. A worker heartbeat is evidence for point 1, never a substitute for the spawner's own check — a stale worker heartbeat (older than two intervals) is treated exactly like silence in point 4.

Default heartbeat interval: the wait's watchdog-tick interval where one is derived, else **10 minutes**, and never longer than the wait's absolute deadline.

A heartbeat interval that passes with nothing observable to report is itself a signal: re-verify the real state instead of assuming the work is still healthy. Rule 2 already governs the rest of the wait's lifecycle — the absolute deadline (point 3) and wake cancellation plus real-state re-verification (point 5) — and applies to every heartbeat wake as well; this rule does not restate them.

## Rule 4 — Time-box every delegation; kill at the deadline.

Every spawned subagent, background command, or delegated task carries a **[time budget]** set by the spawner **before the spawn**, and is killed if it overruns it. A worker with no deadline is an unbounded wait wearing a different name — the same single point of failure Rule 2 forbids.

1. **Set the budget before spawning.**
   - **Default run limit: 1 hour** — applies to any delegation with no better estimate.
   - **Long-running work must be estimated explicitly** before the spawn (model training, large evaluations, data collection/recording, big downloads): from previously recorded durations for similar steps (ledger entries), from the command's own parameters (e.g. steps × observed time-per-step), or from an operator-provided hint. For explicitly estimated work the run limit is **2 × the estimate**.
2. **Arm the deadline.** The Rule 2 time-driven fallback for the accompanying wait must fire **no later than the run limit** — a kill deadline is enforceable only if a wake is guaranteed at it. Record the estimate and the run limit in Rule 2 point 6's pending-wait record.
3. **Kill on overrun.** When the run limit passes and the worker is still running: re-verify real state first (Rule 2 point 5 — it may have finished without waking you), then **kill it** via the platform mechanism *(Claude Code: stop the task / background shell, e.g. `TaskStop` or killing the process. Codex · VS Code Copilot: terminate the worker or abandon the blocking call.)*
4. **Record and re-plan.** A kill is a **timeout blocker**: record the estimate, the actual runtime, and what was killed in the workflow's own record ([loop ledger] entry / [execution report]). Then follow the workflow's blocker path — re-estimate and respawn (never retry with the same estimate unchanged) or restructure the step, then continue the workflow. A killed worker is evidence about the *execution*, never about the artifact being improved — do not count it as a failed or no-progress iteration of the artifact.

## Scope note

This file stays **generic**: it governs engagement, waits, heartbeats, deadlines, wake triggers, and re-verification for every loop and exec workflow alike. Loop-specific control — iteration counters (`max_iterations`, `no_progress_k`), progress accounting, and checkpoint / durable-goal-record semantics — deliberately lives in `_lib/loop_control.md`, not here.
